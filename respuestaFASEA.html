Vista 1 v_resumen_ventas

Muestra cliente, fecha, cantidad de ítems y total. Filtra solo ventas COMPLETADAS. El GROUP BY agrupa todos los detalles de una venta en una sola fila.

DROP VIEW IF EXISTS v_resumen_ventas;
CREATE VIEW v_resumen_ventas AS
SELECT
    c.nombre            AS cliente,
    v.fecha,
    COUNT(d.id_detalle) AS cantidad_productos,
    v.total
FROM venta v
JOIN cliente c       ON v.id_cliente  = c.id_cliente
JOIN detalle_venta d ON v.id_venta    = d.id_venta
WHERE  v.estado = 'COMPLETADA'
GROUP BY v.id_venta, c.nombre, v.fecha, v.total
ORDER BY v.fecha;

-- Prueba:
SELECT * FROM v_resumen_ventas;
-- Deben aparecer 9 filas (10 ventas - 1 ANULADA = 9 COMPLETADAS)
¿Por qué COUNT(d.id_detalle)? Cuenta las filas de detalle_venta asociadas a cada venta. El GROUP BY incluye v.id_venta para que cada venta sea una fila única aunque tenga varios detalles.
Vista 2 v_productos_bajo_stock

Lista los productos cuyo stock actual es menor o igual al stock mínimo definido, con su categoría. Ordenados por los más críticos primero.

DROP VIEW IF EXISTS v_productos_bajo_stock;
CREATE VIEW v_productos_bajo_stock AS
SELECT
    p.nombre       AS producto,
    c.nombre       AS categoria,
    p.stock,
    p.stock_minimo
FROM producto p
JOIN categoria c ON p.id_categoria = c.id_categoria
WHERE p.stock <= p.stock_minimo
ORDER BY (p.stock_minimo - p.stock) DESC;

-- Prueba — resultados esperados con datos iniciales:
-- Corrector liquido   stock=4  minimo=8   (Papeleria)
-- Azucar 1kg          stock=6  minimo=20  (Alimentos)
-- Desinfectante 1L    stock=2  minimo=8   (Limpieza)
ORDER BY (stock_minimo - stock) DESC ordena primero los más críticos. El JOIN con categoria es necesario porque en producto solo existe id_categoria, no el nombre.
Vista 3 v_rentabilidad_productos

Calcula margen en Bs. y porcentaje de margen sobre el costo, ordenado de mayor a menor rentabilidad.

DROP VIEW IF EXISTS v_rentabilidad_productos;
CREATE VIEW v_rentabilidad_productos AS
SELECT
    nombre,
    precio_compra,
    precio_venta,
    (precio_venta - precio_compra)
        AS margen_bs,
    ROUND(((precio_venta - precio_compra) / precio_compra) * 100, 2)
        AS porcentaje_margen
FROM producto
ORDER BY porcentaje_margen DESC;

-- ===== PRUEBA DE ACTUALIZACIÓN EN VIVO =====
-- Paso 1: ver margen actual del Cable USB-C
SELECT nombre, precio_compra, margen_bs
FROM v_rentabilidad_productos
WHERE nombre LIKE '%Cable%';
-- Resultado: margen_bs = 9.50  (18.00 - 8.50)

-- Paso 2: modificar precio_compra SIN recrear la vista
UPDATE producto SET precio_compra = precio_compra + 5 WHERE id_producto = 1;

-- Paso 3: consultar la misma vista
SELECT nombre, precio_compra, margen_bs
FROM v_rentabilidad_productos
WHERE nombre LIKE '%Cable%';
-- Resultado: margen_bs = 4.50  (18.00 - 13.50) ← cambió solo
¿Por qué cambió sin recrear la vista? Una vista no almacena datos. Es un alias de una consulta SQL. Cada vez que ejecutás SELECT * FROM v_rentabilidad_productos, MySQL corre la consulta original en ese instante contra las tablas reales. Como producto ya tiene el nuevo precio_compra, el cálculo devuelve el valor actualizado. No hay caché ni copia.
Experimento Escritura en vistas con JOIN / GROUP BY

Ejecutar cada instrucción por separado y anotar el resultado exacto que devuelve MySQL:

-- Intento 1: UPDATE sobre v_resumen_ventas
UPDATE v_resumen_ventas SET total = 0 WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the UPDATE
-- is not updatable

-- Intento 2: INSERT sobre v_resumen_ventas
INSERT INTO v_resumen_ventas (cliente, fecha, total) VALUES ('Test', NOW(), 0);
-- ERROR 1471 (HY000): The target table v_resumen_ventas of the INSERT
-- is not insertable-into

-- Intento 3: DELETE sobre v_resumen_ventas
DELETE FROM v_resumen_ventas WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the DELETE
-- is not updatable
Los tres intentos fallan. MySQL rechaza escrituras sobre vistas que usan JOIN, GROUP BY o funciones de agregación como COUNT().
Experimento Vista simple sin JOIN ni GROUP BY 
-- Crear vista simple (1 tabla, sin JOIN, sin GROUP BY)
CREATE VIEW v_solo_clientes AS
SELECT id_cliente, nombre, ciudad FROM cliente;

-- UPDATE sobre la vista simple — FUNCIONA:
UPDATE v_solo_clientes SET ciudad = 'Potosi' WHERE id_cliente = 1;
-- Query OK, 1 row affected

-- Verificar en la tabla original:
SELECT ciudad FROM cliente WHERE id_cliente = 1;
-- ciudad = 'Potosi'  ← la tabla real cambió
Respuesta para la demo:

v_resumen_ventas usa JOIN (une 3 tablas) y GROUP BY con COUNT(). MySQL no puede determinar a cuál fila base pertenece un UPDATE, ni cómo desagregar un COUNT. Por eso la bloquea.

v_solo_clientes mapea 1:1 con la tabla cliente — cada fila de la vista corresponde exactamente a una fila de la tabla base.

6 condiciones para que una vista sea actualizable en MySQL:
(1) Sin JOIN (solo una tabla base)  ·  (2) Sin GROUP BY ni HAVING  ·  (3) Sin funciones de agregación (COUNT, SUM, etc.)  ·  (4) Sin DISTINCT  ·  (5) Sin subconsultas en el SELECT  ·  (6) Sin UNION
Trigger 1 trg_valida_stock — BEFORE INSERT

Se ejecuta antes de insertar en detalle_venta. Si la cantidad supera el stock disponible, lanza SIGNAL y cancela la operación.

DROP TRIGGER IF EXISTS trg_valida_stock;

DELIMITER $$
CREATE TRIGGER trg_valida_stock
BEFORE INSERT ON detalle_venta
FOR EACH ROW
BEGIN
    DECLARE v_stock    INT;
    DECLARE v_nombre   VARCHAR(100);

    SELECT stock, nombre
    INTO   v_stock, v_nombre
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    IF NEW.cantidad > v_stock THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = CONCAT(
            'Stock insuficiente para "', v_nombre,
            '". Solicitado: ', NEW.cantidad,
            ', disponible: ', v_stock
        );
    END IF;
END$$
DELIMITER ;

-- ===== PRUEBA =====
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 1, 9999, 18.00, 179982.00);
-- ERROR 1644: Stock insuficiente para "Cable USB-C 1m". Solicitado: 9999, disponible: 40

-- Verificar que NO se insertó nada:
SELECT * FROM detalle_venta ORDER BY id_detalle DESC LIMIT 1;
-- El id_detalle más reciente es anterior al intento → no se guardó
NEW.campo accede al valor que se intentó insertar. El trigger compara con el stock real. SIGNAL SQLSTATE '45000' es el código estándar para errores de usuario en MySQL — aborta el INSERT y propaga el mensaje descriptivo al cliente.
Trigger 2 trg_venta_descuenta_stock — AFTER INSERT

Se ejecuta después de insertar (ya pasó la validación del Trigger 1). Descuenta el stock y registra el movimiento en auditoria_stock.

DROP TRIGGER IF EXISTS trg_venta_descuenta_stock;

DELIMITER $$
CREATE TRIGGER trg_venta_descuenta_stock
AFTER INSERT ON detalle_venta
FOR EACH ROW
BEGIN
    DECLARE v_stock_anterior INT;

    -- Capturar stock ANTES de modificarlo
    SELECT stock INTO v_stock_anterior
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    -- Descontar del inventario
    UPDATE producto
    SET    stock = stock - NEW.cantidad
    WHERE  id_producto = NEW.id_producto;

    -- Registrar auditoría
    INSERT INTO auditoria_stock
        (id_producto, operacion, stock_anterior, stock_nuevo, observacion)
    VALUES (
        NEW.id_producto,
        'VENTA',
        v_stock_anterior,
        v_stock_anterior - NEW.cantidad,
        CONCAT('Venta id: ', NEW.id_venta)
    );
END$$
DELIMITER ;

-- ===== PRUEBA EN ORDEN =====
-- 1. Stock actual del producto 5:
SELECT stock FROM producto WHERE id_producto = 5;   -- stock = 25

-- 2. Insertar venta válida:
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 5, 2, 14.00, 28.00);

-- 3. Verificar que bajó:
SELECT stock FROM producto WHERE id_producto = 5;   -- stock = 23
SELECT * FROM auditoria_stock;
-- operacion=VENTA, stock_anterior=25, stock_nuevo=23

-- ===== PRUEBA BEFORE vs AFTER =====
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 5, 9999, 14.00, 139986.00);
-- ERROR: Trigger 1 (BEFORE) cancela → Trigger 2 (AFTER) NUNCA se ejecuta

SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 3;
-- No hay registro nuevo → confirmado
Orden BEFORE → AFTER: MySQL ejecuta triggers BEFORE primero. Si el BEFORE lanza SIGNAL, la operación se aborta y el AFTER jamás se invoca. Esto garantiza que auditoria_stock solo registra movimientos que realmente ocurrieron.
SP 1 sp_registrar_venta

Registra cabecera + detalle en una sola llamada CALL. Los triggers del Req. 2 se disparan automáticamente al insertar en detalle_venta.

DROP PROCEDURE IF EXISTS sp_registrar_venta;

DELIMITER $$
CREATE PROCEDURE sp_registrar_venta(
    IN  p_id_cliente  INT,
    IN  p_id_producto INT,
    IN  p_cantidad    INT,
    OUT p_resultado   VARCHAR(200)
)
BEGIN
    DECLARE v_precio      DECIMAL(10,2);
    DECLARE v_id_venta    INT;
    DECLARE v_nombre_prod VARCHAR(100);

    -- Captura el SIGNAL del trigger y lo pone en el OUT
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        GET DIAGNOSTICS CONDITION 1 p_resultado = MESSAGE_TEXT;
        ROLLBACK;
    END;

    START TRANSACTION;

    -- 1. Obtener precio y nombre del producto
    SELECT precio_venta, nombre
    INTO   v_precio, v_nombre_prod
    FROM   producto
    WHERE  id_producto = p_id_producto;

    -- 2. Insertar cabecera de venta
    INSERT INTO venta (id_cliente, total, estado)
    VALUES (p_id_cliente, 0, 'COMPLETADA');
    SET v_id_venta = LAST_INSERT_ID();

    -- 3. Insertar detalle → dispara trg_valida_stock y trg_venta_descuenta_stock
    INSERT INTO detalle_venta
        (id_venta, id_producto, cantidad, precio_unitario, subtotal)
    VALUES
        (v_id_venta, p_id_producto, p_cantidad, v_precio, v_precio * p_cantidad);

    -- 4. Actualizar total en cabecera
    UPDATE venta
    SET    total = v_precio * p_cantidad
    WHERE  id_venta = v_id_venta;

    COMMIT;

    SET p_resultado = CONCAT(
        'OK — Venta #', v_id_venta,
        ' | Producto: ', v_nombre_prod,
        ' | Cantidad: ', p_cantidad,
        ' | Total: Bs. ', v_precio * p_cantidad
    );
END$$
DELIMITER ;

-- ===== PRUEBA EXITOSA =====
CALL sp_registrar_venta(1, 4, 3, @res);
SELECT @res;
-- OK — Venta #11 | Producto: Foco LED 9W | Cantidad: 3 | Total: Bs. 22.50

SELECT stock FROM producto WHERE id_producto = 4;
-- bajó en 3 (trigger se disparó automáticamente)

SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 1;
-- operacion=VENTA registrado por el trigger

-- ===== PRUEBA FALLIDA =====
CALL sp_registrar_venta(1, 4, 9999, @res);
SELECT @res;
-- Stock insuficiente para "Foco LED 9W". Solicitado: 9999, disponible: 57
¿Por qué DECLARE EXIT HANDLER? Cuando el trigger lanza SIGNAL, genera una excepción SQL. Sin el handler el SP aborta y el OUT queda vacío. Con el handler, se captura el mensaje del SIGNAL con GET DIAGNOSTICS y se escribe en p_resultado. El ROLLBACK asegura consistencia.

Cadena automática: CALL sp → INSERT detalle_venta → trg_valida_stock (BEFORE) → trg_venta_descuenta_stock (AFTER). Los triggers se activan solos, sin que el SP los llame explícitamente.
SP 2 sp_reporte_ventas_periodo

Consulta ventas en un rango de fechas reutilizando v_resumen_ventas — sin duplicar JOINs.

DROP PROCEDURE IF EXISTS sp_reporte_ventas_periodo;

DELIMITER $$
CREATE PROCEDURE sp_reporte_ventas_periodo(
    IN p_fecha_inicio DATE,
    IN p_fecha_fin    DATE
)
BEGIN
    SELECT *
    FROM   v_resumen_ventas
    WHERE  DATE(fecha) BETWEEN p_fecha_inicio AND p_fecha_fin;
END$$
DELIMITER ;

-- ===== PRUEBAS =====
CALL sp_reporte_ventas_periodo('2025-01-01', '2025-01-31');
-- 3 ventas de enero (ventas 1, 2, 3)

CALL sp_reporte_ventas_periodo('2025-01-01', '2025-03-31');
-- 9 ventas completadas del trimestre
¿Por qué DATE(fecha)? El campo fecha es DATETIME. Sin DATE(), el límite '2025-01-31' se interpreta como '2025-01-31 00:00:00' y excluiría ventas del mediodía del 31. DATE() trunca la hora.

Reutilización: El SP no repite el JOIN ni el WHERE estado='COMPLETADA'. Toda esa lógica ya está en v_resumen_ventas.

Análisis de criterios para EXCELENCIA (90–100)

Criterio 1 — Vistas (90–100)
Las 3 vistas tienen alias descriptivos, filtros correctos y ORDER BY. Se ejecutó la prueba de actualización de precio_compra y se puede explicar que la vista no almacena datos — cada SELECT ejecuta la consulta en vivo sobre las tablas reales. Sin recrearla, el margen cambia automáticamente.
Criterio 2 — Experimento escritura (90–100)
Se documentaron los errores exactos de MySQL (Error 1288 y 1471) sobre v_resumen_ventas. Se comparó con v_solo_clientes que sí acepta escritura. Se explican las 6 condiciones de MySQL para vistas actualizables: sin JOIN, sin GROUP BY/HAVING, sin funciones de agregación, sin DISTINCT, sin subconsultas en SELECT, sin UNION.
Criterio 3 — Triggers (90–100)
Ambos triggers funcionan. El SIGNAL incluye nombre del producto, cantidad solicitada y stock disponible. Se verificó con el experimento BEFORE/AFTER que cuando trg_valida_stock lanza SIGNAL, trg_venta_descuenta_stock nunca se ejecuta y auditoria_stock no registra nada — solo quedan movimientos que realmente ocurrieron.
Criterio 4 — Procedimientos Almacenados (90–100)
sp_registrar_venta usa DECLARE EXIT HANDLER con GET DIAGNOSTICS para capturar el SIGNAL del trigger y devolverlo en el parámetro OUT tanto en éxito como en fallo. sp_reporte_ventas_periodo reutiliza v_resumen_ventas. Se verificó que al insertar en detalle_venta desde el SP, los triggers se disparan automáticamente sin llamarlos explícitamente.
