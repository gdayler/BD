Vista 1 v_resumen_ventas

Por cada venta COMPLETADA: nombre del cliente, fecha, cantidad de productos y total. Ordenado por fecha descendente.

DROP VIEW IF EXISTS v_resumen_ventas;
CREATE VIEW v_resumen_ventas AS
SELECT
    c.nombre            AS cliente,
    v.fecha,
    COUNT(d.id_detalle) AS cantidad_productos,
    v.total
FROM venta v
JOIN cliente c       ON v.id_cliente = c.id_cliente
JOIN detalle_venta d ON v.id_venta   = d.id_venta
WHERE  v.estado = 'COMPLETADA'
GROUP BY v.id_venta, c.nombre, v.fecha, v.total
ORDER BY v.fecha DESC;

-- Prueba:
SELECT * FROM v_resumen_ventas;
-- 9 filas (10 ventas - 1 ANULADA)
¿Por qué GROUP BY incluye v.id_venta? Para que cada venta sea una sola fila aunque tenga múltiples detalles. El COUNT(d.id_detalle) cuenta las líneas de producto de esa venta.
Vista 2 v_productos_bajo_stock

Productos cuyo stock actual es menor o igual al stock mínimo. Incluye nombre, categoría, stock actual y stock mínimo.

DROP VIEW IF EXISTS v_productos_bajo_stock;
CREATE VIEW v_productos_bajo_stock AS
SELECT
    p.nombre       AS producto,
    c.nombre       AS categoria,
    p.stock        AS stock_actual,
    p.stock_minimo
FROM producto p
JOIN categoria c ON p.id_categoria = c.id_categoria
WHERE p.stock <= p.stock_minimo
ORDER BY (p.stock_minimo - p.stock) DESC;

-- Resultados esperados con datos iniciales:
-- Corrector liquido   stock=4   minimo=8   (Papeleria)
-- Desinfectante 1L    stock=2   minimo=8   (Limpieza)
-- Azucar 1kg          stock=6   minimo=20  (Alimentos)
ORDER BY (stock_minimo - stock) DESC pone primero los más críticos. Azúcar tiene diferencia de 14, Corrector 4, Desinfectante 6.
Vista 3 v_rentabilidad_productos

Margen en Bs. y porcentaje de margen por producto, de mayor a menor.

DROP VIEW IF EXISTS v_rentabilidad_productos;
CREATE VIEW v_rentabilidad_productos AS
SELECT
    nombre,
    precio_compra,
    precio_venta,
    (precio_venta - precio_compra)
        AS margen_bs,
    ROUND(((precio_venta - precio_compra) / precio_compra) * 100, 2)
        AS porcentaje_margen
FROM producto
ORDER BY porcentaje_margen DESC;

SELECT * FROM v_rentabilidad_productos;
Esta vista solo usa una tabla (producto), sin JOIN ni GROUP BY. Es la base del Experimento 1 — al cambiar precio_compra, el margen cambia solo porque la vista consulta en vivo.
Vista 4 v_top_clientes

Clientes con mayor total acumulado en ventas completadas. Incluye nombre, ciudad, cantidad de compras y total gastado.

DROP VIEW IF EXISTS v_top_clientes;
CREATE VIEW v_top_clientes AS
SELECT
    c.nombre                  AS cliente,
    c.ciudad,
    COUNT(v.id_venta)          AS cantidad_compras,
    SUM(v.total)               AS total_gastado
FROM cliente c
JOIN venta v ON c.id_cliente = v.id_cliente
WHERE v.estado = 'COMPLETADA'
GROUP BY c.id_cliente, c.nombre, c.ciudad
ORDER BY total_gastado DESC;

SELECT * FROM v_top_clientes;
-- Resultado esperado:
-- Comercial Lopez    Cochabamba   2 compras   Bs. 232.00
-- Tienda El Sol      La Paz       2 compras   Bs. 234.00
-- Distribuidora Norte Santa Cruz  2 compras   Bs. 140.50
-- Minimarket Central Cochabamba   2 compras   Bs. 176.00
-- Juan Perez         Cochabamba   1 compra    Bs. 28.00
GROUP BY c.id_cliente garantiza una fila por cliente. El WHERE estado='COMPLETADA' excluye la venta anulada de María González, por eso no aparece.
Exp 1 Vista se actualiza sola al cambiar precio_compra 
-- Paso 1: ver margen actual
SELECT nombre, precio_compra, margen_bs
FROM v_rentabilidad_productos
WHERE nombre LIKE '%Cable%';
-- nombre: Cable USB-C 1m | precio_compra: 8.50 | margen_bs: 9.50

-- Paso 2: modificar precio_compra (NO recrear la vista)
UPDATE producto SET precio_compra = precio_compra + 5 WHERE id_producto = 1;
-- Query OK, 1 row affected

-- Paso 3: consultar la misma vista sin recrearla
SELECT nombre, precio_compra, margen_bs
FROM v_rentabilidad_productos
WHERE nombre LIKE '%Cable%';
-- nombre: Cable USB-C 1m | precio_compra: 13.50 | margen_bs: 4.50
-- ← El margen cambió de 9.50 a 4.50 sin recrear la vista
¿Por qué cambió solo? Una vista no guarda datos. Es una consulta SQL con nombre. Cada vez que ejecutás SELECT * FROM v_rentabilidad_productos, MySQL ejecuta la consulta original en ese instante contra las tablas reales. No hubo que recrear nada.
Exp 2 Escribir en vista con JOINs — errores exactos de MySQL 
-- Intento UPDATE
UPDATE v_resumen_ventas SET total = 0 WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the UPDATE
--                     is not updatable

-- Intento INSERT
INSERT INTO v_resumen_ventas (cliente, fecha, total) VALUES ('Test', NOW(), 0);
-- ERROR 1471 (HY000): The target table v_resumen_ventas of the INSERT
--                     is not insertable-into

-- Intento DELETE
DELETE FROM v_resumen_ventas WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the DELETE
--                     is not updatable
Los tres fallan. MySQL no puede propagar un UPDATE/DELETE a tablas base cuando la vista usa JOIN o GROUP BY — no sabe a qué fila de qué tabla afectar.
Exp 3 Vista simple vs. vista con JOINs — comparación 
-- Crear vista simple (1 tabla, sin JOIN, sin GROUP BY)
CREATE VIEW v_solo_clientes AS
SELECT id_cliente, nombre, ciudad FROM cliente;

-- UPDATE sobre la vista simple — FUNCIONA:
UPDATE v_solo_clientes SET ciudad = 'Potosi' WHERE id_cliente = 1;
-- Query OK, 1 row affected

-- Verificar en la tabla original:
SELECT ciudad FROM cliente WHERE id_cliente = 1;
-- ciudad = 'Potosi'  ← la tabla real cambió
Respuesta para la demo — condiciones para vista actualizable en MySQL:

(1) Sin JOIN — solo una tabla base  |  (2) Sin GROUP BY ni HAVING  |  (3) Sin funciones de agregación (COUNT, SUM)  |  (4) Sin DISTINCT  |  (5) Sin subconsultas en el SELECT  |  (6) Sin UNION

v_solo_clientes cumple todo → MySQL puede propagar el UPDATE a cliente directamente porque cada fila de la vista corresponde exactamente a una fila de la tabla base.
Trigger 1 trg_valida_stock — BEFORE INSERT

Se ejecuta antes de insertar en detalle_venta. Si la cantidad supera el stock, lanza SIGNAL descriptivo y cancela la operación.

DROP TRIGGER IF EXISTS trg_valida_stock;

DELIMITER $$
CREATE TRIGGER trg_valida_stock
BEFORE INSERT ON detalle_venta
FOR EACH ROW
BEGIN
    DECLARE v_stock  INT;
    DECLARE v_nombre VARCHAR(100);

    SELECT stock, nombre
    INTO   v_stock, v_nombre
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    IF NEW.cantidad > v_stock THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = CONCAT(
            'Stock insuficiente para "', v_nombre,
            '". Pedido: ', NEW.cantidad,
            ', disponible: ', v_stock
        );
    END IF;
END$$
DELIMITER ;

-- Prueba con cantidad inválida:
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 1, 9999, 18.00, 179982.00);
-- ERROR 1644: Stock insuficiente para "Cable USB-C 1m". Pedido: 9999, disponible: 40

-- Verificar que NO se insertó nada:
SELECT * FROM detalle_venta ORDER BY id_detalle DESC LIMIT 1;
-- El último id_detalle es anterior al intento → el INSERT fue cancelado

-- ¿La auditoría registró algo?
SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 3;
-- No hay registro nuevo → BEFORE canceló antes de que AFTER llegara a ejecutarse
NEW.campo accede al valor que se intenta insertar. SIGNAL SQLSTATE '45000' es el código estándar de MySQL para errores de usuario — aborta el INSERT y propaga el mensaje.
Trigger 2 trg_venta_descuenta_stock — AFTER INSERT

Se ejecuta después del INSERT exitoso en detalle_venta. Descuenta el stock y registra en auditoria_stock.

DROP TRIGGER IF EXISTS trg_venta_descuenta_stock;

DELIMITER $$
CREATE TRIGGER trg_venta_descuenta_stock
AFTER INSERT ON detalle_venta
FOR EACH ROW
BEGIN
    DECLARE v_stock_anterior INT;

    -- Leer stock antes de modificarlo
    SELECT stock INTO v_stock_anterior
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    -- Descontar inventario
    UPDATE producto
    SET    stock = stock - NEW.cantidad
    WHERE  id_producto = NEW.id_producto;

    -- Registrar movimiento en auditoría
    INSERT INTO auditoria_stock
        (id_producto, operacion, stock_anterior, stock_nuevo, observacion)
    VALUES (
        NEW.id_producto,
        'VENTA',
        v_stock_anterior,
        v_stock_anterior - NEW.cantidad,
        CONCAT('Venta id: ', NEW.id_venta)
    );
END$$
DELIMITER ;

-- Prueba — ejecutar en orden:
SELECT stock FROM producto WHERE id_producto = 5;  -- stock = 25

INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 5, 2, 14.00, 28.00);

SELECT stock FROM producto WHERE id_producto = 5;  -- stock = 23 ← bajó
SELECT * FROM auditoria_stock;
-- operacion=VENTA, stock_anterior=25, stock_nuevo=23, observacion='Venta id: 1'
Orden de ejecución BEFORE → AFTER: MySQL ejecuta BEFORE primero. Si el BEFORE lanza SIGNAL, la operación se cancela y el AFTER nunca llega a ejecutarse. Por eso auditoria_stock solo contiene movimientos reales.
Trigger 3 trg_compra_aumenta_stock — AFTER INSERT

Se ejecuta después de insertar en detalle_compra. Aumenta el stock y registra con operación COMPRA.

DROP TRIGGER IF EXISTS trg_compra_aumenta_stock;

DELIMITER $$
CREATE TRIGGER trg_compra_aumenta_stock
AFTER INSERT ON detalle_compra
FOR EACH ROW
BEGIN
    DECLARE v_stock_anterior INT;

    -- Leer stock antes de modificarlo
    SELECT stock INTO v_stock_anterior
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    -- Aumentar inventario
    UPDATE producto
    SET    stock = stock + NEW.cantidad
    WHERE  id_producto = NEW.id_producto;

    -- Registrar movimiento en auditoría
    INSERT INTO auditoria_stock
        (id_producto, operacion, stock_anterior, stock_nuevo, observacion)
    VALUES (
        NEW.id_producto,
        'COMPRA',
        v_stock_anterior,
        v_stock_anterior + NEW.cantidad,
        CONCAT('Compra id: ', NEW.id_compra)
    );
END$$
DELIMITER ;

-- Prueba:
SELECT stock FROM producto WHERE id_producto = 14;  -- stock = 2

INSERT INTO compra (id_proveedor, total) VALUES (2, 70.00);
INSERT INTO detalle_compra (id_compra, id_producto, cantidad, precio_unitario, subtotal)
VALUES (LAST_INSERT_ID(), 14, 20, 3.50, 70.00);

SELECT stock FROM producto WHERE id_producto = 14;  -- stock = 22 ← subió
SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 1;
-- operacion=COMPRA, stock_anterior=2, stock_nuevo=22
Diferencia con trg_venta_descuenta_stock: La tabla objetivo es detalle_compra (no detalle_venta), el stock sube con stock + NEW.cantidad, y la operación en auditoría es 'COMPRA'.
SP 1 sp_registrar_venta

Registra cabecera + detalle en una sola llamada CALL. Si el trigger rechaza la operación, el error llega al parámetro OUT.

DROP PROCEDURE IF EXISTS sp_registrar_venta;

DELIMITER $$
CREATE PROCEDURE sp_registrar_venta(
    IN  p_id_cliente  INT,
    IN  p_id_producto INT,
    IN  p_cantidad    INT,
    OUT p_resultado   VARCHAR(200)
)
BEGIN
    DECLARE v_precio      DECIMAL(10,2);
    DECLARE v_id_venta    INT;
    DECLARE v_nombre_prod VARCHAR(100);

    -- Captura el SIGNAL del trigger y lo devuelve en el OUT
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        GET DIAGNOSTICS CONDITION 1 p_resultado = MESSAGE_TEXT;
        ROLLBACK;
    END;

    START TRANSACTION;

    -- 1. Obtener precio y nombre
    SELECT precio_venta, nombre
    INTO   v_precio, v_nombre_prod
    FROM   producto
    WHERE  id_producto = p_id_producto;

    -- 2. Cabecera de venta
    INSERT INTO venta (id_cliente, total, estado)
    VALUES (p_id_cliente, 0, 'COMPLETADA');
    SET v_id_venta = LAST_INSERT_ID();

    -- 3. Detalle → aquí se disparan trg_valida_stock y trg_venta_descuenta_stock
    INSERT INTO detalle_venta
        (id_venta, id_producto, cantidad, precio_unitario, subtotal)
    VALUES
        (v_id_venta, p_id_producto, p_cantidad, v_precio, v_precio * p_cantidad);

    -- 4. Actualizar total en cabecera
    UPDATE venta
    SET    total = v_precio * p_cantidad
    WHERE  id_venta = v_id_venta;

    COMMIT;

    SET p_resultado = CONCAT(
        'OK — Venta #', v_id_venta,
        ' | ', v_nombre_prod,
        ' x', p_cantidad,
        ' | Total: Bs. ', v_precio * p_cantidad
    );
END$$
DELIMITER ;

-- ===== PRUEBA EXITOSA =====
CALL sp_registrar_venta(1, 4, 3, @res);
SELECT @res;
-- OK — Venta #11 | Foco LED 9W x3 | Total: Bs. 22.50

SELECT stock FROM producto WHERE id_producto = 4;
-- bajó 3 unidades → el trigger se disparó automáticamente

SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 1;
-- operacion=VENTA registrado por el trigger, no por el SP

-- ===== PRUEBA FALLIDA =====
CALL sp_registrar_venta(1, 4, 9999, @res);
SELECT @res;
-- Stock insuficiente para "Foco LED 9W". Pedido: 9999, disponible: 57
¿Cómo llega el error del trigger al OUT? El DECLARE EXIT HANDLER FOR SQLEXCEPTION captura la excepción generada por el SIGNAL. GET DIAGNOSTICS CONDITION 1 extrae el mensaje y lo escribe en p_resultado. El ROLLBACK deja la BD en estado consistente.

¿Los triggers se disparan desde el SP? Sí. El trigger no distingue si el INSERT viene de un usuario directo o de un SP — se activa en cualquier INSERT sobre detalle_venta.
SP 2 sp_reporte_ventas_periodo

Devuelve ventas completadas en un rango de fechas. Reutiliza v_resumen_ventas — no duplica JOINs.

DROP PROCEDURE IF EXISTS sp_reporte_ventas_periodo;

DELIMITER $$
CREATE PROCEDURE sp_reporte_ventas_periodo(
    IN p_fecha_inicio DATE,
    IN p_fecha_fin    DATE
)
BEGIN
    SELECT *
    FROM   v_resumen_ventas
    WHERE  DATE(fecha) BETWEEN p_fecha_inicio AND p_fecha_fin;
END$$
DELIMITER ;

-- ===== PRUEBAS =====
CALL sp_reporte_ventas_periodo('2025-01-01', '2025-03-31');
-- 9 ventas completadas del trimestre (excluye la ANULADA de marzo)

CALL sp_reporte_ventas_periodo('2025-01-01', '2025-01-31');
-- 3 ventas de enero únicamente
¿Por qué DATE(fecha)? El campo fecha es DATETIME. Sin DATE(), el límite '2025-01-31' se convierte a '2025-01-31 00:00:00' y excluiría ventas del mediodía. DATE() trunca la hora. La lógica de filtrado solo COMPLETADAS ya está en v_resumen_ventas, no hay que repetirla.
