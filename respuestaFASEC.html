Prerequisito: Los objetos del Grupo B (v_resumen_ventas, v_productos_bajo_stock, trg_valida_stock, trg_venta_descuenta_stock, sp_registrar_venta) deben estar creados antes de ejecutar los objetos adicionales de esta sección.
Vista adicional v_top_clientes

Los 5 clientes con mayor total acumulado en ventas COMPLETADAS. Incluye nombre, ciudad, cantidad de compras y total gastado.

DROP VIEW IF EXISTS v_top_clientes;
CREATE VIEW v_top_clientes AS
SELECT
    c.nombre                  AS cliente,
    c.ciudad,
    COUNT(v.id_venta)          AS cantidad_compras,
    SUM(v.total)               AS total_gastado
FROM cliente c
JOIN venta v ON c.id_cliente = v.id_cliente
WHERE v.estado = 'COMPLETADA'
GROUP BY c.id_cliente, c.nombre, c.ciudad
ORDER BY total_gastado DESC
LIMIT 5;

SELECT * FROM v_top_clientes;
-- Resultado esperado (con datos iniciales):
-- Tienda El Sol        La Paz       2   234.00
-- Comercial Lopez      Cochabamba   2   232.00
-- Minimarket Central   Cochabamba   2   176.00
-- Distribuidora Norte  Santa Cruz   2   140.50
-- Juan Perez           Cochabamba   1    28.00
-- (Maria Gonzalez no aparece: su única venta está ANULADA)
GROUP BY c.id_cliente garantiza una fila por cliente aunque tenga múltiples ventas. El LIMIT 5 va en la definición de la vista para que siempre devuelva el top 5. El WHERE estado='COMPLETADA' excluye la venta anulada — María González no aparece porque su única venta es ANULADA.
Trigger adicional trg_producto_inactivo — BEFORE INSERT

Se ejecuta antes de insertar en detalle_venta. Si el producto tiene activo = 0, lanza SIGNAL con el nombre del producto en el mensaje.

DROP TRIGGER IF EXISTS trg_producto_inactivo;

DELIMITER $$
CREATE TRIGGER trg_producto_inactivo
BEFORE INSERT ON detalle_venta
FOR EACH ROW
BEGIN
    DECLARE v_activo  TINYINT(1);
    DECLARE v_nombre  VARCHAR(100);

    -- SELECT dentro del trigger para obtener nombre y estado
    SELECT activo, nombre
    INTO   v_activo, v_nombre
    FROM   producto
    WHERE  id_producto = NEW.id_producto;

    IF v_activo = 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = CONCAT(
            'El producto [', v_nombre, '] no esta disponible.'
        );
    END IF;
END$$
DELIMITER ;

-- ===== PRUEBA =====
-- Primero desactivar el producto:
UPDATE producto SET activo = 0 WHERE id_producto = 2;

-- Intentar vender ese producto:
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 2, 1, 75.00, 75.00);
-- ERROR 1644 (45000): El producto [Teclado inalambrico] no esta disponible.

-- Restaurar para no afectar otras pruebas:
UPDATE producto SET activo = 1 WHERE id_producto = 2;
¿Cómo se incluye el nombre en el mensaje? Se hace un SELECT ... INTO v_nombre FROM producto WHERE id_producto = NEW.id_producto dentro del cuerpo del trigger, antes del IF. Luego se usa CONCAT() para armar el mensaje dinámico. El mensaje resultante incluye el nombre real del producto entre corchetes: El producto [Teclado inalambrico] no esta disponible.

Orden de ejecución en detalle_venta: Si el producto está inactivo, trg_producto_inactivo (BEFORE) lanza SIGNAL y cancela todo — trg_valida_stock (BEFORE) y trg_venta_descuenta_stock (AFTER) nunca se ejecutan. En MySQL, múltiples triggers del mismo evento/momento se ejecutan en el orden en que fueron creados.
SP adicional sp_productos_reponer

Sin parámetros. Usa v_productos_bajo_stock y agrega el nombre del último proveedor que proveyó cada producto. Si nunca fue comprado, muestra 'Sin proveedor'.

DROP PROCEDURE IF EXISTS sp_productos_reponer;

DELIMITER $$
CREATE PROCEDURE sp_productos_reponer()
BEGIN
    SELECT
        bs.producto,
        bs.categoria,
        bs.stock_actual,
        bs.stock_minimo,
        IFNULL(prov.nombre, 'Sin proveedor') AS ultimo_proveedor
    FROM v_productos_bajo_stock bs
    -- Subconsulta: obtener el id del último proveedor por producto
    LEFT JOIN (
        SELECT
            dc.id_producto,
            pr.nombre
        FROM detalle_compra dc
        JOIN compra c   ON dc.id_compra    = c.id_compra
        JOIN proveedor pr ON c.id_proveedor = pr.id_proveedor
        WHERE (dc.id_producto, c.fecha) IN (
            -- Fecha más reciente de compra por cada producto
            SELECT dc2.id_producto, MAX(c2.fecha)
            FROM  detalle_compra dc2
            JOIN  compra c2 ON dc2.id_compra = c2.id_compra
            GROUP BY dc2.id_producto
        )
    ) prov ON bs.producto = (
        SELECT nombre FROM producto WHERE id_producto = prov.id_producto
    )
    ORDER BY (bs.stock_minimo - bs.stock_actual) DESC;
END$$
DELIMITER ;

-- ===== ALTERNATIVA MÁS LIMPIA con JOIN directo a producto =====
DROP PROCEDURE IF EXISTS sp_productos_reponer;

DELIMITER $$
CREATE PROCEDURE sp_productos_reponer()
BEGIN
    SELECT
        p.nombre                          AS producto,
        cat.nombre                        AS categoria,
        p.stock                           AS stock_actual,
        p.stock_minimo,
        IFNULL(ult.nombre_proveedor, 'Sin proveedor') AS ultimo_proveedor
    FROM producto p
    JOIN categoria cat ON p.id_categoria = cat.id_categoria
    -- LEFT JOIN para preservar productos sin compras ("Sin proveedor")
    LEFT JOIN (
        -- Subconsulta: último proveedor por producto
        SELECT
            dc.id_producto,
            pr.nombre AS nombre_proveedor
        FROM detalle_compra dc
        JOIN compra co    ON dc.id_compra    = co.id_compra
        JOIN proveedor pr ON co.id_proveedor = pr.id_proveedor
        WHERE (dc.id_producto, co.fecha) IN (
            SELECT dc2.id_producto, MAX(co2.fecha)
            FROM  detalle_compra dc2
            JOIN  compra co2 ON dc2.id_compra = co2.id_compra
            GROUP BY dc2.id_producto
        )
    ) ult ON p.id_producto = ult.id_producto
    -- Misma condición que v_productos_bajo_stock
    WHERE p.stock <= p.stock_minimo
    ORDER BY (p.stock_minimo - p.stock) DESC;
END$$
DELIMITER ;

-- ===== PRUEBA =====
CALL sp_productos_reponer();
-- Resultado esperado (datos iniciales):
-- Azucar 1kg         Alimentos  6    20   Alimentos Del Valle
-- Desinfectante 1L   Limpieza   2     8   Sin proveedor  ← nunca se compró
-- Corrector liquido  Papeleria  4     8   Papelera del Sur
¿Por qué LEFT JOIN y no INNER JOIN? Si se usa INNER JOIN, los productos sin ninguna compra en detalle_compra desaparecerían del resultado. Con LEFT JOIN, cuando la subconsulta no encuentra proveedor, el campo queda NULL y IFNULL() lo convierte en 'Sin proveedor'.

Lógica del "último proveedor": La subconsulta interna calcula MAX(fecha) por producto en compra. La condición WHERE (id_producto, fecha) IN (...) filtra solo la fila más reciente de compra por producto, garantizando que se muestre el último proveedor, no todos.
Exp A Vista inválida silenciosa

Se renombra una columna en la tabla base sin modificar la vista. MySQL no avisa al momento del ALTER — solo falla cuando se consulta.

-- Paso 1: Crear la vista (ya existe del Grupo B, pero la redefinimos aquí)
DROP VIEW IF EXISTS v_resumen_ventas;
CREATE VIEW v_resumen_ventas AS
    SELECT c.nombre AS cliente, v.fecha, COUNT(d.id_detalle) AS cant, v.total
    FROM venta v JOIN cliente c ON v.id_cliente=c.id_cliente
    JOIN detalle_venta d ON v.id_venta=d.id_venta
    WHERE v.estado='COMPLETADA' GROUP BY v.id_venta;
-- Query OK — vista creada sin errores

-- Paso 2: Renombrar la columna en la tabla base (SIN tocar la vista)
ALTER TABLE venta RENAME COLUMN total TO monto_total;
-- Query OK — MySQL NO avisa nada. El ALTER se ejecuta sin error.
-- La vista v_resumen_ventas sigue "existiendo" en el catálogo pero ya es inválida.

-- Paso 3: Consultar la vista
SELECT * FROM v_resumen_ventas;
-- ERROR 1054 (42S22): Unknown column 'v.total' in 'field list'
-- ← El error aparece AQUÍ, no en el paso 2

-- Paso 4: Restaurar para no romper el resto de la clase
ALTER TABLE venta RENAME COLUMN monto_total TO total;
SELECT * FROM v_resumen_ventas;
-- 9 filas — vuelve a funcionar correctamente
¿En qué paso exacto falló? En el paso 3 — la consulta a la vista. El paso 2 (ALTER TABLE) se ejecutó sin error ni advertencia.

¿Por qué MySQL no avisa en el paso 2? MySQL almacena las vistas como texto SQL en el catálogo del sistema (information_schema.VIEWS). Al crear la vista guarda solo el texto del SELECT, no verifica dependencias inversas. El motor no lleva un registro de qué vistas usan qué columnas de qué tablas, por lo que al renombrar una columna no sabe que debe invalidar algo. Solo cuando se intenta ejecutar la vista MySQL parsea el SQL almacenado contra la estructura real de las tablas y detecta que la columna ya no existe.
Implicación en un sistema productivo: Este comportamiento es extremadamente peligroso. Una migración de base de datos (renombrar columna, cambiar tipo, eliminar tabla) puede silenciosamente romper decenas de vistas, SPs y triggers que nadie va a notar hasta que se consulten en producción. Las estrategias para prevenirlo son: (1) usar CHECK TABLE v_resumen_ventas después de cada ALTER, (2) mantener un inventario de dependencias entre objetos, (3) ejecutar una suite de pruebas automáticas sobre las vistas críticas después de cualquier migración, (4) en MySQL 8+ usar information_schema.VIEW_TABLE_USAGE para rastrear dependencias antes de hacer cambios.
Exp B Escritura directa en vistas — JOINs vs vista simple

Se compara el comportamiento de DML (UPDATE/INSERT/DELETE) sobre una vista compleja con JOINs vs. una vista simple de una sola tabla.

-- ===== PARTE 1: DML sobre v_resumen_ventas (JOINs + GROUP BY) =====

UPDATE v_resumen_ventas SET total = 99999 WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the UPDATE is not updatable

INSERT INTO v_resumen_ventas (cliente, fecha, total) VALUES ('Test', NOW(), 0);
-- ERROR 1471 (HY000): The target table v_resumen_ventas of the INSERT is not insertable-into

DELETE FROM v_resumen_ventas WHERE 1=1;
-- ERROR 1288 (HY000): The target table v_resumen_ventas of the DELETE is not updatable
-- CONCLUSIÓN: los 3 DML fallan sobre una vista con JOIN + GROUP BY

-- ===== PARTE 2: Vista simple (1 tabla, sin JOIN, sin agregación) =====
CREATE VIEW v_solo_clientes AS
SELECT id_cliente, nombre, ciudad FROM cliente;

-- UPDATE — FUNCIONA:
UPDATE v_solo_clientes SET ciudad = 'Potosi' WHERE id_cliente = 1;
-- Query OK, 1 row affected

-- Verificar en tabla real:
SELECT ciudad FROM cliente WHERE id_cliente = 1;
-- ciudad = 'Potosi'  ← la tabla base cambió, no solo la vista

-- DELETE — FUNCIONA:
DELETE FROM v_solo_clientes WHERE id_cliente = 6;
-- Query OK, 1 row affected
-- La fila desaparece también de la tabla cliente

-- INSERT sobre vista simple — NO funciona si falta columna NOT NULL sin DEFAULT:
INSERT INTO v_solo_clientes (nombre, ciudad) VALUES ('Nuevo', 'Cbba');
-- Query OK (funciona si la tabla acepta NULL o tiene DEFAULT en otras columnas)

-- Restaurar cliente 1:
UPDATE v_solo_clientes SET ciudad = 'Cochabamba' WHERE id_cliente = 1;
¿Qué condiciones necesita MySQL para que una vista sea actualizable?

La vista debe cumplir todas estas condiciones simultáneamente:
  (1) Sin JOIN — solo referencia a una tabla base  |  (2) Sin GROUP BY ni HAVING  |  (3) Sin funciones de agregación (COUNT, SUM, etc.)  |  (4) Sin DISTINCT  |  (5) Sin subconsultas en el SELECT  |  (6) Sin UNION  |  (7) Sin ORDER BY (en versiones estrictas)
Implicación práctica: Las vistas con JOIN son de solo-lectura en MySQL. Permitir DML directo sobre vistas complejas podría generar inconsistencias (¿en qué tabla base se propaga el INSERT con columnas de dos tablas?). Por eso MySQL lo prohíbe. En un sistema productivo: nunca exponer vistas complejas como actualizables — usar SPs para las operaciones de escritura, o crear INSTEAD OF triggers (no disponibles en MySQL, sí en PostgreSQL/Oracle).
Exp C Vista que referencia a otra vista — dependencia en cadena

Se crea una vista sobre otra vista, y luego se elimina la vista base. MySQL no verifica dependencias al hacer DROP.

-- Paso 1: Crear vista derivada (sobre otra vista)
CREATE VIEW v_ventas_grandes AS
SELECT * FROM v_resumen_ventas WHERE total > 100;
-- Query OK — vista creada sin problemas

-- Paso 2: Consultar la vista derivada — funciona bien
SELECT * FROM v_ventas_grandes;
-- Muestra ventas con total > 100 (3 o 4 filas)

-- Paso 3: Eliminar la vista BASE (la que usa v_ventas_grandes)
DROP VIEW v_resumen_ventas;
-- Query OK — MySQL NO avisa nada. No hay protección por dependencias.

-- Paso 4: Consultar la vista derivada
SELECT * FROM v_ventas_grandes;
-- ERROR 1356 (HY000): View 'bd_ventas.v_ventas_grandes' references invalid table(s)
--                     or column(s) or function(s) or definer/invoker of view
--                     lack rights to use them
-- ← El error ocurre aquí, no en el DROP

-- Restaurar para el resto de la clase:
CREATE VIEW v_resumen_ventas AS
    SELECT c.nombre AS cliente, v.fecha, COUNT(d.id_detalle) AS cant, v.total
    FROM venta v JOIN cliente c ON v.id_cliente=c.id_cliente
    JOIN detalle_venta d ON v.id_venta=d.id_venta
    WHERE v.estado='COMPLETADA' GROUP BY v.id_venta;

SELECT * FROM v_ventas_grandes;
-- Vuelve a funcionar correctamente
¿MySQL avisó algo al hacer el DROP? No. El DROP VIEW se ejecutó sin ningún error ni advertencia, a pesar de que v_ventas_grandes depende de v_resumen_ventas. Esto es el mismo comportamiento silencioso del Experimento A: MySQL no mantiene un grafo de dependencias entre objetos del catálogo.
Implicación y cómo prevenirlo: En sistemas productivos con capas de vistas (vistas sobre vistas), eliminar o modificar cualquier vista base puede romper silenciosamente toda la cadena. Estrategias preventivas: (1) antes de hacer DROP, consultar SELECT * FROM information_schema.VIEW_TABLE_USAGE WHERE TABLE_NAME = 'v_resumen_ventas' para detectar dependencias, (2) documentar explícitamente el árbol de dependencias, (3) establecer una política de "no vistas sobre vistas" en producción para reducir la superficie de riesgo, (4) ejecutar un script de validación de todas las vistas después de cualquier cambio de esquema.
Exp D BEFORE cancela el AFTER — orden de ejecución de triggers

Con trg_valida_stock (BEFORE) y trg_venta_descuenta_stock (AFTER) activos, se intenta insertar una cantidad mayor al stock.

-- Estado previo:
SELECT stock FROM producto WHERE id_producto = 1;
-- stock = 40
SELECT COUNT(*) FROM auditoria_stock;
-- Supongamos 0 registros previos

-- Intentar insertar con cantidad inválida (9999 > 40):
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 1, 9999, 18.00, 179982.00);
-- ERROR 1644 (45000): Stock insuficiente para "Cable USB-C 1m". Pedido: 9999, disponible: 40

-- ¿La auditoría registró algo?
SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 3;
-- (empty set) — NO hay nuevo registro en auditoria_stock

-- ¿El stock cambió?
SELECT stock FROM producto WHERE id_producto = 1;
-- stock = 40 — SIGUE IGUAL. No cambió nada.
Diagrama del flujo BEFORE → INSERT → AFTER:
INSERT → [1] trg_valida_stock (BEFORE) detecta 9999 > 40 → lanza SIGNAL → [2] El INSERT es cancelado (la fila nunca se escribe) → [3] trg_venta_descuenta_stock (AFTER) nunca se ejecuta

¿Por qué el SIGNAL detiene todo? Un SIGNAL en un trigger BEFORE genera una excepción que aborta la sentencia DML completa antes de que la fila sea escrita. Los triggers AFTER solo se ejecutan cuando el evento (INSERT/UPDATE/DELETE) fue exitoso. Si el BEFORE falla, no hay evento exitoso, por lo tanto el AFTER jamás se dispara.
Exp E Trigger disparado desde dentro de un SP

Se llama a sp_registrar_venta con cantidad válida e inválida. Se verifica si el trigger se dispara desde el SP y si el error llega al parámetro OUT.

-- ===== LLAMADA CON CANTIDAD VÁLIDA =====
CALL sp_registrar_venta(1, 5, 2, @res);
SELECT @res;
-- @res = 'OK — Venta #11 | Destornillador set x2 | Total: Bs. 28.00'
-- ← El SP terminó con éxito y devolvió mensaje en el OUT

-- ¿Se disparó el trigger desde el SP?
SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 3;
-- operacion=VENTA, id_producto=5, observacion='Venta id: 11'
-- ← SÍ, el trigger AFTER se disparó automáticamente desde el INSERT del SP

SELECT stock FROM producto WHERE id_producto = 5;
-- stock bajó 2 unidades — el trigger actuó aunque fue llamado desde el SP

-- ===== LLAMADA CON CANTIDAD INVÁLIDA =====
CALL sp_registrar_venta(1, 5, 9999, @res);
SELECT @res;
-- @res = 'Stock insuficiente para "Destornillador set". Pedido: 9999, disponible: 23'
-- ← El mensaje del SIGNAL del trigger llegó al parámetro OUT del SP

-- ¿La auditoría registró algo nuevo?
SELECT * FROM auditoria_stock ORDER BY id_auditoria DESC LIMIT 3;
-- Sin nuevos registros — BEFORE canceló, AFTER no se ejecutó
-- El stock tampoco cambió
¿El trigger se disparó aunque el INSERT vino del SP? Sí, siempre. Los triggers son mecanismos del motor de base de datos que se ejecutan como respuesta a eventos sobre tablas (INSERT, UPDATE, DELETE). No importa el origen del evento — usuario directo, SP, función, o incluso otro trigger. Cuando el SP hace INSERT INTO detalle_venta, el motor dispara los triggers asociados a esa tabla exactamente igual que si el INSERT viniera de una conexión directa.

¿Cómo llegó el error del trigger al OUT del SP? El SP tiene DECLARE EXIT HANDLER FOR SQLEXCEPTION. Cuando el trigger lanza SIGNAL, MySQL genera una excepción SQL que el handler captura. GET DIAGNOSTICS CONDITION 1 p_resultado = MESSAGE_TEXT extrae el texto del SIGNAL y lo escribe en el parámetro OUT. El ROLLBACK deja la BD consistente.
Exp F Trigger con mensaje dinámico — SELECT dentro del trigger

Se demuestra cómo hacer un SELECT dentro del trigger para incluir el nombre del producto en el mensaje de error. Desafío de la guía completado.

-- Primero, desactivar el producto id=2:
UPDATE producto SET activo = 0 WHERE id_producto = 2;

-- Intentar vender el producto inactivo:
INSERT INTO detalle_venta (id_venta, id_producto, cantidad, precio_unitario, subtotal)
VALUES (1, 2, 1, 75.00, 75.00);
-- ERROR 1644 (45000): El producto [Teclado inalambrico] no esta disponible.
-- ← El nombre del producto está incluido en el mensaje

-- ===== CLAVE DEL DESAFÍO: cómo se hace SELECT dentro del trigger =====
-- En el cuerpo del trigger (dentro de BEGIN...END) se usa:
DECLARE v_nombre VARCHAR(100);

SELECT nombre INTO v_nombre
FROM   producto
WHERE  id_producto = NEW.id_producto;
-- NEW.id_producto contiene el valor del campo que se está intentando insertar
-- La variable v_nombre queda con el nombre real del producto
-- Luego se usa en CONCAT() para armar el mensaje dinámico

-- Restaurar:
UPDATE producto SET activo = 1 WHERE id_producto = 2;
¿Cómo se hace SELECT dentro de un trigger? Igual que dentro de un SP: se declara una variable con DECLARE y se usa SELECT ... INTO variable. El trigger tiene acceso completo al contexto SQL: puede leer cualquier tabla, usar funciones, declarar variables locales. El valor NEW.id_producto está disponible porque el trigger BEFORE se ejecuta antes de que la fila se escriba, pero después de que los valores del INSERT sean conocidos.

Los triggers respetan el contexto del evento: El trigger no distingue ni le importa si el INSERT provino de una instrucción directa, de un SP, de otro trigger, o de una aplicación conectada via JDBC/PDO. El evento es el evento: en el momento en que MySQL procesa un INSERT sobre detalle_venta, los triggers BEFORE y AFTER se disparan siempre, sin excepción.

Parte 4 — El sistema como un todo

Flujo completo documentado paso a paso. Para cada paso se indica qué objeto intervino, si fue llamado explícitamente o se disparó solo, y la justificación de negocio.

1 Consultar v_productos_bajo_stock Vista — llamada explícita 
SELECT * FROM v_productos_bajo_stock;
-- Resultado:
-- Azucar 1kg         Alimentos  6    20  ← diferencia 14, el más crítico
-- Desinfectante 1L   Limpieza   2     8  ← diferencia 6
-- Corrector liquido  Papeleria  4     8  ← diferencia 4
Objeto: Vista v_productos_bajo_stock | Activación: Explícita (SELECT directo). La vista ejecuta su consulta en tiempo real — no hay datos almacenados, refleja el estado actual de la tabla producto.
2 Vender el stock actual de Azúcar (6 unidades) SP + Triggers 
CALL sp_registrar_venta(1, 12, 6, @res);
SELECT @res;
-- @res = 'OK — Venta #12 | Azucar 1kg x6 | Total: Bs. 16.80'
Objetos en cadena:
① sp_registrar_venta — llamado explícitamente, inserta en venta y detalle_venta.
② trg_producto_inactivo (BEFORE) — disparado automáticamente. Azúcar está activo → pasa sin error.
③ trg_valida_stock (BEFORE) — disparado automáticamente. 6 ≤ 6 → pasa.
④ trg_venta_descuenta_stock (AFTER) — disparado automáticamente. Descuenta 6 unidades → stock queda en 0. Registra en auditoria_stock.
3 Consultar v_productos_bajo_stock de nuevo Vista — llamada explícita 
SELECT * FROM v_productos_bajo_stock;
-- Resultado:
-- Azucar 1kg         Alimentos  0    20  ← aparece con stock=0
-- Desinfectante 1L   Limpieza   2     8
-- Corrector liquido  Papeleria  4     8
¿Aparece con stock=0? Sí. La vista no tiene datos propios — siempre consulta la tabla producto en tiempo real. Como el trigger del paso 2 ya actualizó el stock a 0, la vista refleja ese valor inmediatamente. La condición stock <= stock_minimo sigue siendo verdadera (0 ≤ 20), por lo tanto Azúcar sigue apareciendo.
4 Intentar vender 1 unidad más de Azúcar SP + Trigger BEFORE 
CALL sp_registrar_venta(1, 12, 1, @res);
SELECT @res;
-- @res = 'Stock insuficiente para "Azucar 1kg". Pedido: 1, disponible: 0'
-- ← El trigger BEFORE bloqueó la operación y el mensaje llegó al OUT
Objeto que actuó: trg_valida_stock (BEFORE INSERT en detalle_venta), disparado automáticamente desde el SP. El mensaje de error es dinámico (incluye nombre y stock). El SP capturó el SIGNAL con su handler y lo devolvió en el OUT.
5 Consultar auditoria_stock Tabla — llamada explícita 
SELECT * FROM auditoria_stock ORDER BY id_auditoria;
-- Hay exactamente 1 registro (solo la venta exitosa del paso 2):
-- id=1 | id_producto=12 | operacion=VENTA | stock_anterior=6 | stock_nuevo=0
--      | observacion='Venta id: 12'
-- El intento fallido del paso 4 NO generó registro (BEFORE lo cancela antes del AFTER)
¿Qué dice la auditoría? Solo registra movimientos reales de stock. El intento fallido del paso 4 no aparece porque el BEFORE canceló la operación antes de que el AFTER pudiera ejecutarse. Esto garantiza que auditoria_stock solo contiene hechos reales del negocio, nunca intentos fallidos.
6 Llamar sp_productos_reponer SP — llamada explícita 
CALL sp_productos_reponer();
-- Resultado:
-- Azucar 1kg         Alimentos  0    20   Alimentos Del Valle  ← aparece con stock=0
-- Desinfectante 1L   Limpieza   2     8   Sin proveedor
-- Corrector liquido  Papeleria  4     8   Papelera del Sur
¿Aparece Azúcar con stock=0? Sí. El SP consulta la condición stock <= stock_minimo en tiempo real sobre la tabla producto, igual que la vista. Además muestra el último proveedor que lo suministró (Alimentos Del Valle) y el Desinfectante — que nunca tuvo compra en detalle_compra — muestra correctamente 'Sin proveedor' gracias al LEFT JOIN con IFNULL.

Análisis de criterios para EXCELENCIA (90–100)

Criterio 2 — Ruptura de vistas A, B, C (90–100)
Los 3 experimentos ejecutados con resultados anotados y explicaciones que demuestran comprensión del comportamiento del motor. Además se documentó la implicación práctica de cada uno en un sistema productivo con estrategias concretas de prevención:
Exp A (vista inválida silenciosa): MySQL no avisa en el ALTER TABLE (paso 2). El error 1054 aparece en el paso 3 (SELECT). Implicación: migraciones pueden romper vistas sin advertencia. Prevención: CHECK TABLE + suite de pruebas post-migración + consultar information_schema.VIEW_TABLE_USAGE.
Exp B (escritura en vistas): Error 1288 en UPDATE/DELETE y 1471 en INSERT sobre v_resumen_ventas (JOINs). UPDATE y DELETE funcionan en v_solo_clientes (tabla simple). Las 6 condiciones de MySQL para vistas actualizables documentadas. Implicación: usar SPs para escritura, nunca exponer vistas complejas como actualizables.
Exp C (dependencia entre vistas): DROP VIEW sin advertencia. Error 1356 al consultar la vista dependiente. Implicación: árbol de dependencias invisible al motor. Prevención: information_schema.VIEW_TABLE_USAGE, política de no-vistas-sobre-vistas en producción.

Los 3 experimentos completos con explicaciones que demuestran comprensión profunda del comportamiento del motor:
Exp D (BEFORE cancela AFTER): Con cantidad 9999 > stock=40, el BEFORE lanza SIGNAL, cancela el INSERT, el AFTER nunca se ejecuta, auditoria_stock no recibe registro nuevo y el stock no cambia. El diagrama INSERT → BEFORE → [cancelado] → AFTER[no ejecutado] está documentado.
